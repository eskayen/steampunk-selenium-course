'KISS' and 'DRY' Code

Content

Ideally in the software world, code should be simple, easy to understand, and less resistant to change. Imagine you have a program that is thousands or tens of thousands of lines of code, and you realize you have to change something early on in the code that will require you to change possibly the thousands of lines of code that come after it or depend upon what you changed.

'KISS' and 'DRY' are two very important concepts to understand in the software development field. 'KISS' stands for "Keep It Simple, Stupid" and 'DRY' stands for "Don't Repeat Yourself."

'KISS' mainly emphasizes that you do not need to and should not make code more complex than it needs to be, because in the long run it can cause more harm than good.

'DRY' is a principle that simply states that you do not need to write same code repeatedly.

These principles apply to any coding/software development environment, whether it is web development, automated testing (Selenium), making video games, etc...

We are going to use Java in this module, but these principles apply to any programming language. Let's take a look at code that fails to adhere to the 'DRY' principle:

import io.github.bonigarcia.wdm.WebDriverManager;
import org.openqa.selenium.By;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.WebElement;

import org.openqa.selenium.support.ui.WebDriverWait;
import org.testng.annotations.Test;

import static org.testng.Assert.assertEquals;

public class SampleCode {

    @Test
    public void search_for_wikipedia(){
        WebDriverManager.chromedriver().setup();
        WebDriver driver = new ChromeDriver();
        driver.get("https://www.coveros.com");

        WebElement searchBox = driver.findElement(By.id("s"));
        searchBox.sendKeys("wikipedia");
        searchBox.submit();

        new WebDriverWait(driver, 10)
                .until(ExpectedConditions.visibilityOfElementLocated(By.className("header-blog")));
        assertEquals(driver.getTitle(), "You searched for wikipedia - Coveros");

        driver.quit();
    }

    @Test
    public void search_for_cheese(){
        WebDriverManager.chromedriver().setup();

        driver.get("https://www.coveros.com");

        WebElement searchBox = driver.findElement(By.id("s"));
        searchBox.sendKeys("cheese");
        searchBox.submit();

        new WebDriverWait(driver, 10)
                .until(ExpectedConditions.visibilityOfElementLocated(By.className("header-blog")));
        assertEquals(driver.getTitle(), "You searched for cheese - Coveros");

        driver.quit();
    }

    @Test
    public void search_for_selenium() {

        WebDriverManager.chromedriver().setup();
        WebDriver driver = new ChromeDriver();
        driver.get("https://www.coveros.com");

}

Essentially, the code is the same as what we used in the "Integrating Selenium into Java" portion of the course, except its 3 tests that search for cheese, wikipedia, and selenium respectively.

The code passes successfully, however, if it was to be graded on the 'DRY' principle, it would receive an F-.

You can see that many parts are repeating: such as setting up, searching for something and submitting it, and even just quitting webdriver. This can be a problem because when you have thousands of lines of code (or thousands of tests), if you realize there is something that needs to be changed in any one of fields, you would have to reiterate through each line and it can be a really tedious process.

Usually automated testing is implemented after development is finished but there is still a chance that a lone developer changes a single id of an important WebElement, and this can lead to catastrophic failures in our test scripts. Code that follows the 'DRY' principle would make it easier for us to correct the test scripts incase this hypothetical situation happens (very rare).

So let's look back at our code and try and find out what we can "fix". We notice that webdriver is set up in every single method, so that is something we can make another method for.

Now with a new method to set up webdriver, our code looks like this:

import io.github.bonigarcia.wdm.WebDriverManager;
import org.openqa.selenium.By;
import org.openqa.selenium.WebDriver;

}

WebDriver driver; is declared outside of the methods since it will use that variable throughout all the tests. This new set_up(url) method allows us to also easily alter the setup process for all of the tests, instead of going by each test one at a time. It also allows us to add Options, for example headless, which will discussed later on.

Keep in mind that the methods such as set_up are not actual tests, so they do not have the @Test annotation.

Also keep in mind that, in my case, the methods verify_result and search_for_text need to have the same String parameter, otherwise it will fail. Be careful in the logic of what makes your test case pass/fail.

But more can be added, or actually reduced. Let's say that this class contains searchbox tests, so I am going to be using the same searchBox element multiple times, and I am going to verifying the title multiple times. So I will create methods for those both so I can reduce the repeating code in those tests. Now our entire code looks like this:

import io.github.bonigarcia.wdm.WebDriverManager;
import org.openqa.selenium.By;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.WebElement;
import org.openqa.selenium.chrome.ChromeDriver;
import org.openqa.selenium.support.ui.ExpectedConditions;
import org.openqa.selenium.support.ui.WebDriverWait;
import org.testng.annotations.Test;

import static org.testng.Assert.assertEquals;

public class SampleCode {
    WebDriver driver;

    public void set_up(String url){
        WebDriverManager.chromedriver().setup();
        driver = new ChromeDriver();
        driver.get(url);
    }

    WebElement searchBox;

    public void search_for_text(String text){
        WebElement searchBox = driver.findElement(By.id("s"));
        searchBox.sendKeys(text);
        searchBox.submit();
    }

    public void verify_result(String expected_result){
        new WebDriverWait(driver, 10)
                .until(ExpectedConditions.visibilityOfElementLocated(By.className("header-blog")));
        assertEquals(driver.getTitle(), "You searched for " + expected_result + " - Coveros");
    }
    
    @Test
    public void search_for_wikipedia(){
        set_up("https://www.coveros.com");
        search_for_text("wikipedia");
        verify_result("wikipedia");
        driver.quit();
    }

    @Test
    public void search_for_cheese(){
        set_up("https://www.coveros.com");
        search_for_text("cheese");
        verify_result("cheese");
        driver.quit();
    }

    @Test
    public void search_for_selenium() {
        set_up("https://www.coveros.com");
        search_for_text("selenium");
        verify_result("selenium");
        driver.quit();
    }
}

Now each test is only 6 lines of code (7 including the @Test), and if were phenomenom were to happen like an ID changing, we can just go back to the search_for_text method and just change it from there instead of going back to each line.

Another thing you may have noticed is that driver.quit() repeats alot. We have not made a method for that yet, and we can if we wanted to. For example, if I wanted to have Selenium take screenshots of each test case and save them, or if I wanted to do some extra stuff before quitting driver, I can make another method for all that stuff and append it to the end of each test case: 

public void tear_down(){
        //do extra stuff here
        driver.quit();
    }

So now our 'improved' code looks like this:

import org.openqa.selenium.By;

import org.openqa.selenium.support.ui.WebDriverWait;

import static org.testng.Assert.assertEquals;

public class SampleCode {
    WebDriver driver;

    public void set_up(String url){
        WebDriverManager.chromedriver().setup();
        driver = new ChromeDriver();
        driver.get(url);
    }

    WebElement searchBox;

    public void search_for_text(String text){
        WebElement searchBox = driver.findElement(By.id("s"));
        searchBox.sendKeys(text);
        searchBox.submit();
    }

    public void verify_result(String expected_result){
        new WebDriverWait(driver, 10)
                .until(ExpectedConditions.visibilityOfElementLocated(By.className("header-blog")));
        assertEquals(driver.getTitle(), "You searched for " + expected_result + " - Coveros");
    }

    public void tear_down(){
        //do extra stuff here
        driver.quit();
    }

    @Test
    public void search_for_wikipedia(){
        set_up("https://www.coveros.com");
        search_for_text("wikipedia");
        verify_result("wikipedia");
        tear_down();
    }

    @Test
    public void search_for_cheese(){
        set_up("https://www.coveros.com");
        search_for_text("cheese");

        search_for_text("selenium");
        verify_result("selenium");

    }
}

Screenshotting test cases and saving them is for another module, and we can include that in our tear_down instead of repeating it at the end of each test.

Another very important feature that will be discussed are Before and After methods. These can and will make your code more elegant, straight-forward, and easy to understand.
